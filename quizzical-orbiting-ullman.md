# 週日の集会割当ツール 実装計画

## Context

`C:\develop\週日の集会割当\要件定義.md` (v2, 2026-07-06 確定) に基づく新規開発。
JW週日集会の担当割り当てを、履歴に基づく優先度つきプルダウンで支援するデスクトップアプリ。
操作者は非技術者の責任者1名。成果物は名前入りExcel（元テンプレのE列へ書き戻し）。

- 既存コードなし（グリーンフィールド）。リポジトリ未初期化 → `git init` から始める
- 検証用実データ: `G:\マイドライブ\jw.org\徳山会衆\週日の集会割当\2026年7-8月\2026年7-8月.xlsx`（構造解析済み、要件定義 §11）
- 環境: Node v24 / npm 11 あり。**Rust/cargo なし → rustup 導入が前提**

## 確定済み方針（要件定義 v2 より）

- **Tauri v2** デスクトップアプリ（Win/Mac）、フロントは **Vite + React + TS**（ユーザー確認済み）
- 保存 = **IndexedDB + data.json 二重書き込み**、JSON手動バックアップも提供
- Excel読み書き = **ExcelJS**（書式・結合セル保持）、1シート2週分
- 優先度 = ロール（または優先度グループ）単位の最終担当日昇順、未担当者最優先、
  タイブレーク=通算回数→ランダム。相手役はペア履歴（主）+個人履歴（副）
- 進め方 = **ウォーキングスケルトン優先**（ユーザー確認済み）: Excel取込→1日割当→Excel書き戻しを最短で1本通す

## プロジェクト構成

```
C:\develop\週日の集会割当\app\        ← Tauriプロジェクトルート
├─ src\                     # React フロント（ロジックは全てここ）
│  ├─ domain\               # 純粋ロジック（UI/IO非依存、単体テスト対象）
│  │  ├─ types.ts           # Member/Role/RoleGroup/PriorityGroup/Meeting/Slot/Assignment/PairHistory
│  │  ├─ priority.ts        # 優先度エンジン（候補生成・ソート・タイブレーク・ペア履歴）
│  │  ├─ catalog.ts         # プログラム型カタログ（行→型→スロット構成の解決、型シグネチャ記憶）
│  │  └─ nameMatch.ts       # 名寄せ（フルネーム完全一致→曖昧候補提示）
│  ├─ io\
│  │  ├─ excelImport.ts     # ExcelJS: 2週分シート分割→行分類→Meeting/Slot生成
│  │  ├─ excelExport.ts     # ExcelJS: 元ワークブックへE列書き戻し→別名保存
│  │  ├─ pdfImport.ts       # pdfjs-dist: 過去PDF→テキスト抽出→履歴レコード
│  │  └─ storage.ts         # Dexie(IndexedDB) + data.json 二重書き込み・復旧・手動バックアップ
│  ├─ screens\              # 画面（§7 画面一覧に対応）
│  │  ├─ AssignScreen.tsx   # 割り当て（1画面=1集会日）
│  │  ├─ MemberScreen.tsx   # 成員マスター
│  │  ├─ RoleScreen.tsx     # ロール/ロールグループ/優先度グループ
│  │  ├─ ImportScreen.tsx   # Excel取込＋レビュー確定
│  │  ├─ HistoryImportScreen.tsx  # PDF履歴インポート＋名寄せ
│  │  └─ ExportScreen.tsx   # エクスポート
│  └─ App.tsx               # ルーティング（タブ切替で十分、Router不要）
├─ src-tauri\               # Rust薄シェル（fs/dialogプラグイン設定のみ、独自Rustコードほぼなし）
└─ tests\                   # vitest（domain層の単体テスト）
```

主要依存: `exceljs` / `pdfjs-dist` / `dexie` / `zustand`（軽量状態管理）/ `vitest`
Tauriプラグイン: `@tauri-apps/plugin-fs`（data.json読み書き）, `@tauri-apps/plugin-dialog`（ファイル選択・保存先）

**data.json の置き場所**: 既定は実行ファイル隣接ディレクトリ（フォルダコピーでPC移行、要件§3）。
Mac .app や書き込み不可の場所では `appDataDir` へフォールバックし、画面に実パスを表示する。

## マイルストーン

### M0: 足場（0.5日）
1. rustup で Rust 導入（MSVC toolchain + WebView2 確認）
2. `npm create tauri-app@latest` → Vite+React+TS テンプレート、`app\` に配置、`git init`
3. `storage.ts`: Dexie スキーマ + data.json 二重書き込み + 起動時復旧ロジック（IndexedDB空→JSONから復元）
4. vitest セットアップ、タブ切替の空画面5枚
- **検証**: `npm run tauri dev` でウィンドウが開き、ダミーデータが再起動後も残り、data.json にも書かれている

### M1: ウォーキングスケルトン（1〜2日）★リスク検証の山場
Excel取込 → 1日分割当（素朴な候補表示）→ E列書き戻し → 別名保存、を実データで1本通す。
1. `excelImport.ts`: 実ワークブック読込 → シートを日付行(A列がDate)で2回分に分割 →
   行分類（C列キーワード/セクション見出し/part番号/E列ラベル併用、要件§4.3）→ Meeting+Slot 生成。
   型カタログは**ハードコード初期値**（会衆聖書研究=司会+朗読の2枠、実演=生徒+相手 等）から始める
2. `AssignScreen.tsx`: 集会日選択 → スロット一覧 → プルダウン（この段階では在籍成員全員・並び順なし）→ 保存
3. `excelExport.ts`: **元ファイルを ExcelJS で開き直して**該当セルのE列ラベル後ろに名前追記（敬称なし、`/`区切り）→ 別名保存
4. 成員はJSON直編集 or 最小限の追加フォームで10名程度投入
- **検証（重要・実ファイルで）**:
  - 2026年7-8月.xlsx の8回分が正しい日付・スロット構成で取り込める（巡回週08-18の「奉仕の話」が非割当扱いになる）
  - 書き戻したファイルを Excel で開き、**書式・結合セル・列幅が崩れていない**こと（ExcelJSの書式保持検証、要件§10）
  - 開会の祈り行（テンプレ追加済み）の書き戻し先セルを最新ファイルで確認

### M2: 優先度エンジン（1〜2日）
1. `types.ts` 完成: Role/PriorityGroup/Assignment/PairHistory + 派生データ（最終担当日・通算回数・ペア最終日）
2. `priority.ts`: 候補生成（ロール保持×在籍）→ ソート（最終担当日昇順、未担当最優先、通算回数→ランダム）
   → 相手役はペア最終日（主）+個人最終日（副）、既定同性フィルタ+「全員表示」トグル
3. AssignScreen へ統合: 優先度順プルダウン、同一日重複の確認ダイアログ、保存=履歴確定、
   保存済み日の再オープン→修正→再保存（旧割当を履歴から取り消して上書き）
4. **vitest で優先度ロジックを重点的に単体テスト**（未担当最優先/グループ合算/ペア履歴/タイブレーク/再保存の履歴整合）
- **検証**: テスト green + 手動で「保存→次週の並びが変わる」ことを確認

### M3: マスター管理画面（1日）
1. MemberScreen: 氏名/性別/状態、ロール・ロールグループ付与（状態変更で候補から外れる、削除なし）
2. RoleScreen: ロール/ロールグループ/優先度グループの定義・編集
3. 初期データ投入: 既定ロール一式 + 優先度グループ初期値（宝part1+2、生活の討議・話）を seed として組込み
- **検証**: ロール付与→割当画面の候補プールに即反映、優先度グループ変更→並び順が変わる

### M4: 取込レビュー + PDF履歴インポート（1〜2日）
1. ImportScreen: 自動検出結果のレビューUI（行ごとに型/ロール/スロット数を確認・修正、未分類行の明示、
   part6「話」の相手役省略）→ 修正を**型シグネチャ単位で記憶**し次回自動適用
2. `pdfImport.ts`: **最初に実PDFを1つ読んでテキスト抽出の行・列順を検証してから**パーサ確定（要件§10）。
   抽出 → Excelと同じ型解決を流用 → E列相当の名前を取り出し
3. `nameMatch.ts` + HistoryImportScreen: フルネーム完全一致は自動、曖昧のみ選択UI（対応を記憶）
- **検証**: 過去3か月の実PDFから履歴が入り、割当画面の並び順に反映される

### M5: 仕上げ・配布（1日 + Mac CI）
1. 手動バックアップ（JSON書き出し/読み込み）、終了前の未保存警告
2. エクスポート画面（対象月の全日付をワークブックへ一括書き戻し→保存ダイアログ）
3. Windows ビルド（`npm run tauri build`）→ 実機で SmartScreen 回避手順確認
4. GitHub Actions で macOS ビルド（リポジトリを GitHub へ push、tauri-action 使用）
5. 操作者向け1枚マニュアル（起動・取込・割当・エクスポート・バックアップ）
- **検証**: ビルド済み .exe を別フォルダへコピー→データごと動く（フォルダコピー移行の確認）

## リスクと対策（早期に潰す順）

| リスク | 対策 |
|---|---|
| ExcelJS が実テンプレの書式を壊す | M1 で最初に実ファイル往復を検証。壊れる場合は「セル値のみ書き換え」に限定する実装へ |
| PDF テキスト抽出の行・列順崩れ | M4 冒頭で実PDF検証をパーサ実装前に実施 |
| 型自動検出の誤判定 | レビュー確定UIが安全弁。未分類は明示して無視/手動分類 |
| exe隣接への data.json 書き込み不可（権限） | appDataDir フォールバック + 実パス表示 |
| Rust 環境構築のつまずき | M0 で最初に実施。WebView2 は Win11 標準搭載で問題なし |

## 検証方法（全体）

- domain 層: vitest 単体テスト（優先度エンジン中心）
- IO 層: 実ワークブック `2026年7-8月.xlsx` と過去実PDFでの手動往復テスト
- E2E: M1 完了時点から「取込→割当→書き戻し→Excelで目視」を各マイルストーンで反復
- 最終: ビルド済みバイナリでのフォルダコピー移行テスト
